Configuring democratic-csi

democratic-csi is a Kubernetes CSI driver that turns a ZFS box into dynamically-provisioned PersistentVolumes. Its zfs-generic-nvmeof driver provisions a zvol per PVC and exports it over NVMe-oF — which is exactly what this appliance serves. Point it at this host and greendot stays the dashboard: every CSI-created subsystem shows up on Exports badged External with a live RDMA dot, so you can see at a glance whether a volume is honestly on RDMA or has fallen back to TCP.

1. Worker-node prerequisites

Each Kubernetes worker that mounts a volume needs the NVMe fabric modules loaded, plus an RDMA-capable NIC (or Soft-RoCE) if you want the green dot:

modprobe nvme nvme-tcp nvme-rdma nvme-fc
# persist across reboots:
printf 'nvme\nnvme-tcp\nnvme-rdma\nnvme-fc\n' > /etc/modules-load.d/nvme.conf

The node plugin runs nvme connect -t rdma …; without nvme-rdma and a working RDMA interface it silently falls back to TCP — which is precisely the case the green dot exists to catch.

2. SSH access — read this first

democratic-csi's zfs-generic-* drivers do not use a CSI API on the box. They open an SSH session and run zfs and nvmetcli directly, as root (or a sudo-capable user). That is a root-equivalent credential held by the cluster, and it bypasses greendot's helper and its typed allowlist entirely. Treat the key accordingly. If your threat model can't accept that, don't run democratic-csi here — drive provisioning through greendot's helper instead.

3. Driver config

A zfs-generic-nvmeof config, adapted from the upstream example (the authoritative list of options lives there):

driver: zfs-generic-nvmeof
sshConnection:
  host: STORAGE_HOST            # this greendot box
  port: 22
  username: root               # or a sudo-capable user
  privateKey: |                # a key the cluster holds (or use `password:`)
    -----BEGIN OPENSSH PRIVATE KEY-----
    ...
zfs:
  datasetParentName: tank/k8s/vols          # one zvol per PVC, created here
  detachedSnapshotsDatasetParentName: tank/k8s/snaps
nvmeof:
  shareStrategy: nvmetCli
  transports:
    - "rdma://STORAGE_HOST:4420"   # rdma:// is the whole RDMA switch; use tcp:// for TCP
  nvmetCli:
    basename: "nqn.2003-01.org.linux-nvme"
  subsystem:
    attributes:
      allow_any_host: 1           # SEE FENCING NOTE BELOW

Fencing. allow_any_host: 1 is the example default and means any initiator can attach — the opposite of what RWO single-writer safety needs. For a real SAN, lock each subsystem to the consuming node's NQN (allowed_hosts) so a rescheduled pod can't leave two nodes writing the same block device.

4. Snapshots

greendot's own scheduled snapshots and democratic-csi's CSI snapshots are the same kind of object — native ZFS copy-on-write — so they coexist cleanly. The behaviour is a per-VolumeSnapshotClass toggle:

# VolumeSnapshotClass parameters
detachedSnapshots: "false"          # false (default) = native `zfs snapshot`  (instant, COW, ~0 space)
                                    # true            = `zfs send | recv`      (independent, full copy)
detachedVolumesFromSnapshots: "false"   # false = `zfs clone`; true = full send/recv copy

Keep snapshot naming distinct from greendot's scheduled snapshots so neither's retention prunes the other's.

5. Coexistence with greendot

  • NVMe-oF is safe. greendot only reconciles subsystems under its own NQN prefix and leaves foreign ones untouched, so CSI-created subsystems are never reaped — they just appear on Exports as External.
  • Do not run democratic-csi's iSCSI driver on this box. greendot applies iSCSI by rendering the whole config and running targetctl restore, which would clobber targetcli-created targets on the next reconcile. Use NVMe-oF here.
